Command categories
------------------

Want to clear this up because it's been in the back of my head for ages - commands will groupable based on a few properties, and if i can get it right I'll have a better chance of creating a nice consistent syntax.

* polar or cartesian (or other) - what coordinate types the arguments are
* absolute or relative - compared to the page or the current turtle position
* drawing or non-drawing - does the command add any drawing elements to the page?
* moving or non-moving - does the command change the turtles position?
* rotating or non-rotating - does the command change the turtle's heading?

For example if some move or draw commands could be made absolute or relative, or non-drawing (or whatever) with the use of certain modifiers that might be quite nice.
Modifiers could be flags, capitals, prefixes/suffixes etc.
Also some combos may or may not make sense, want to explore.

### Polar and cartesian, absolute and relative

Polar: left, right, bear, jump
Cartesian: move

So for example if any of these could be made non-drawing with a modifier, then jump would go away.
Also absolute versions could be made to go to certain spots on the page.
In that instance 'move' would be badly named, but 'movexy' 'xy' or 'cartesian' might be better.

For modifiers you could even use stock arithmetic operators to emphasize the maths of it:

```
xy	200,400		// Absolute - go to those coords
+xy 200,400		// relative - add those coords to the current
-xy 200,400		// relative - subtract from the current
```
That would be quite cool, but it would kind of imply that anything without the prefix was absolute.
But then there's more issues - are they also relative to turtle's heading...?
What about:
```
xy = 200,400	// absolute - go to these cartesian coords
xy + 200,400	// relative - add these coords to the current
xy - 200,400	// relative - subtract from the current

ra = 200,45		// absolute - go to these polar coords
ra + 200,45		// relative - add these polar coords (cf `bear`)
ra - 200,45		// relative - subtract these polar coords
```
The second last being nearly the same as the current `bear` command, except the order swapped. ( You could add an `ar` if needed too.)
Whether 'radius' or 'distance' makes more sense for the last two is also up for debate.
If I decide to use shorthands like these there'll be a land-grab for characters and a fight over things like degrees/distance, radius/radians.
Some of that could be handled with less-short shorthands:
```
rdeg = 200, 45
rrad = 200, 0.78
rpi  = 200, 0.25
rtau = 200, 0.125
```

Using arithmetic operators would tend to rule out having them in command or identifier names, so probably no 'rect-rounded' or anything like that unless spaces were enforced.
There's enough here that it's starting to look interesting.

Single-valued commands
----------------------

To simplify things I want to focus on the idea of single-valued commands for a while, such as just operating on x or y, and work up from there.

A speculative `x` command like this could have a few interpretations:
```
	x 100
```
* Set the turtle's absolute/page position.x to 100; a horizontal move from the user's perspective. This maps all position inputs to single vertical line.
* Add 100 to the turtle's absolute/page position.x; also a horizontal move. Over all position inputs this is an x-translate operation.
* Add 100 to x from *the turtle's perspective*. Akin to a sideways 'strafe' move in gaming. An x & y translation.
* Set the turtle's frame.x to 100 - exactly the same as above as he's always at (0,0,0) in that perspective

Similarly for `y 100`:
* Set the turtle's absolute/page position.y to 100, ie move vertically from the user's perspective. This maps all position inputs to single horizontal line.
* Add 100 to the turtle's absolute/page position.y, also a vertical move. Over all position inputs this is a y-translate operation.
* Add 100 to y from *the turtle's perspective*. Equiv to a forward (or backward) move. An x & y translation.
* Set the turtle's frame.y to 100 - again, the same as addition

So for "x" and "y" we'd need three syntactic variants.
* page absolute
* page relative
* frame relative/absolute

Let's do polar radius and angle as well to see if they work out similarly:

`radius 100`
* Set the turtle's absolute/page position.radius to 100. An inward or outward radial move that maps all positions to a circle
* Add 100 to the turtle's absolute/page radius (or distance from the origin).
	Move the turtle outward (or inward) from the origin from the user's perspective
	If multiplied and the input & output signs match, would be equivalent to a scale ie zoom/shrink. If the output signs change it's a mirror as well.
	As an addition it's not an affine transformation, it's some kind of warp.
* Add 100 to the radius from the turtle's frame. This is basically just a forward move.
* Again set works out the same as add

`angle 100`
* Set the turtle absolute/page angle to 100 (a rotation). Maps all position inputs to single radial line.
* Add 100 to the turtle's absolute/page angle (angle from the polar axis). Equivalent to a rotate transformation.
* Add 100 to the angle from the turtle's frame. A rotation on the spot.
* Set the turtle's frame angle to 100, again the same as addition.

Syntactic variants
------------------

For x, y, radius and angle the three syntactic variant needed would be:
* Page absolute
* Page relative
* Frame (turtle) relative/absolute - these work out to be the same

So given argument `arg`:

| property	| page absolute			| page relative			| turtle frame			|
| --------	| -------------			| -------------			| ------------			|
| x			| turtle.x = arg		| turtle.x += arg		| 'strafe' arg			|
| y			| turtle.y = arg		| turtle.y += arg		| forward/backward arg	|
| radius	| turtle.polar.r = arg	| turtle.polar.r += arg	| forward/backward arg	|
| angle		| turtle.polar.a = arg  | turtle.polar.a += arg	| rotate arg			|

That's fairly good so far.
There are few extra details that would need to be sorted though.
* The angle and 'strafe' (whatever it gets called) commands would probably need left, right and relative-to-polar-direction versions, like the bear commands have now.
* The angle operations end up the same in page relative and turtle frame modes

### Turtle Rotation
Also, so far this has avoided the question of whether any of these commands also imply or include turtle rotations.
Some of them could conceivably, but it would be like an extra feature that you could turn on, a bit like the xyTurn command I have now (or the no-draw prefix).
More generally the feature would be something like: 'do whatever command, then turn the turtle to point along the line of the movement'.

Shortly I'll start breaking down some of the existing commands into a kind of grammar, and the directional turn will get something like an `@`.

### Location and Position
Not going to go full longhand for these right now, but a little bit of thought seems to suggest that the results will be pretty similar for the 2 and 3 arg versions as well.
Ie there will be the same three variants -  page abs, page rel, and turtle frame.

### Page Absolute Syntax

For page absolute settings you might want to indicate assignment by adding an equals into the command:
```
	x=		100
	y=		200
	radius= 500
	angle=	50
```

Note at this point we're still using a statement structure like `[turtle implied] $command $args` so signifying assignment has to be part of the command. We don't have any proper operators yet.
(Also the no-draw variants might look a bit weird: `~y= 200` anyone?
I might have to mandate space after the tilde modifier to make it clearer.)

This sort of structure would work well for location and position as well, eg `xy= 200,100`.

It also suggests syntax for page relative in things like `x+ 300` `y- 100`

### Page relative and turtle frame syntax


I'm not sure yet how to distinguish between page relative and turtle frame, so I might need to revise the page variants to capitals such as `X= 100`, `X+ 100`.
(NB this would different to SVG conventions).
That would leave lowercase to be strictly turtle-frame specific.
